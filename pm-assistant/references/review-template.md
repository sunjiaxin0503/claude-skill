# 需求文档审查报告模板

以下是审查报告的标准结构。审查的核心态度：作为产品经理最严厉的父亲，不放过任何一个可能导致开发返工、线上事故、或用户体验问题的疏漏。

---

| 审查日期 | YYYY-MM-DD |
| 审查版本 | 对应 PRD 版本号 |
| 审查结论 | 🔴 不通过 / 🟡 有条件通过 / 🟢 通过 |

## 审查总结

[2-3 句话概括整体评价，先说优点再说问题]

### 问题统计

| 🟡 中等 | X | 建议修改，不改可能导致返工 |

## 5. 找茬环节

这一部分是刻意站在最挑剔的角度来审视需求文档。

### 从不同角色视角的挑战

#### 开发者视角
[如果我是接手这个需求的开发，我最想吐槽什么？]

#### 测试者视角
[如果我是负责测试的 QA，我会觉得哪些地方测试用例写不出来？]

#### 运营者视角
[如果我是使用这个功能的运营人员，我会觉得哪里不好用？]

#### 终端用户视角
[如果我是最终用户，我的使用体验会在哪里卡住？]

### 需求本身的挑战
- [这个需求真的有必要做吗？有没有更简单的替代方案？]
- [MVP 的范围是否合理？会不会砍得太多失去价值，或者太大难以交付？]
- [半年后回头看，这个设计会不会过时或需要大改？]

## 审查心态指南

审查时保持以下心态：

1. **不要手软**：这是你写的 PRD，但审查时你要假装从没见过这个需求，用全新的、挑剔的眼光来看
2. **问「如果...怎么办」**：对每个功能描述，连续问三个「如果...怎么办」，往往能发现边界情况
3. **想象最懒的开发**：如果开发只按字面意思来实现，不做任何合理推断，文档还够用吗？
4. **想象最刁钻的测试**：QA 会怎么测？有没有写不出测试用例的模糊地带？
5. **想象上线后的第一个投诉**：用户最可能在哪里遇到问题？
