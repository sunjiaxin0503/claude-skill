# 游戏运营活动需求文档编写 Skill

## 概述

本 skill 用于通过逐步提问的方式，帮助用户（游戏运营策划、产品经理、研发、测试）编写完整的游戏运营活动需求文档。最终输出为 Markdown 格式的中文文档。

## 核心工作流程

采用**分阶段交互式提问**的方式收集信息，共分为 5 个阶段。每个阶段收集完毕后，给用户简短确认，再进入下一阶段。在提问时使用 `ask_user_input_v0` 工具呈现选项，开放性问题用自然语言提问。

**重要原则**：
- 每轮最多提问 3 个问题，避免一次性提问过多导致用户疲劳
- 有默认值或可推导的信息不必反复确认
- 如果用户在某个问题上给出了超出选项的详细描述，直接采纳，不要强制回到选项框架
- 用户可能在一开始就给出大量信息（如完整的活动方案），此时应跳过已回答的问题，只补问缺失部分

---

## 阶段 1：基础信息收集

收集活动的元信息和基本定义。

**必问项：**

1. **关联游戏/项目名称**：哪个游戏项目？（开放性问题）
2. **活动名称**：这个活动叫什么？（开放性问题）
3. **活动背景描述**：为什么要做这个活动？（如新版本上线、节日运营、拉新促活等）
4. **预期活动目标**：量化目标是什么？（如预约人数、参与率、回流率等）
5. **预期对外时间**：活动计划什么时候上线？持续多久？
6. **需求方/负责人信息**：策划负责人是谁？

---

## 阶段 2：平台与用户范围

确定活动的技术边界和目标用户。

**必问项：**

1. **活动上线平台**（可多选）：
   - 网页（H5）
   - 微信小程序
   - 游戏内嵌页（WebView）

   **关键联动**：平台选择直接决定登录方式，在文档中自动填充：
   - 网页 → 通过世游通行证账号登录
   - 小程序 → 手机号授权登录世游通行证
   - 内嵌页 → 游戏账号静默登录（Combo ID）

2. **参与用户范围**（可多选）：
   - 官网用户（仅世游通行证注册用户）
   - 联运用户（通过联运渠道进入的用户）
   - 全渠道用户（所有用户均可参与）

   **联运用户特别说明**：联运用户登录游戏使用的是渠道账号（如华为账号、小米账号），在网页和小程序中参与活动时，无法通过世游通行证账号直接找到对应的游戏账号。因此，当活动涉及发放游戏道具奖励或获取游戏内任务进度时，联运用户需先完成**游戏账号（Combo ID）关联世游通行证**的流程。关联完成后，后续参与活动即可通过世游通行证找到游戏账号（Combo ID）。如果参与用户范围包含联运用户或全渠道用户，且活动涉及游戏道具发放或游戏内数据读取，文档中必须包含 Combo ID 关联流程说明。

3. **参与维度**：
   - 用户 ID 级别（世游通行证 ID）
   - 角色 ID 级别（游戏内角色维度）

   如果涉及游戏道具发放，务必确认参与维度。

## 阶段 3：活动玩法设计

这是需求文档的核心部分。一个活动可包含多个玩法组合。

**不要直接向用户展示玩法列表让其选择**，因为用户可能不了解 event 系统的玩法分类。正确做法是：

1. **请用户用自然语言描述活动的参与流程**：「用户进来之后做什么？怎么玩？怎么获得奖励？」
2. **根据用户描述，自动识别并推荐对应的系统玩法**，向用户解释「你描述的这个环节，在我们的活动系统中对应 XX 玩法」
3. **逐一确认**，确保用户理解每个玩法的含义，并补问该玩法所需的细节配置

**无法匹配已有玩法时的处理流程**：

如果用户描述的某个活动环节无法明确对应到上述任何一种已有玩法，或者不确定应该使用哪种玩法来实现，需按以下步骤处理：

1. **主动告知用户**：「你描述的这个环节，在现有的活动玩法中没有直接对应的类型，可能需要确认是否可以用现有玩法变通实现，或者需要新增玩法。」
2. **尝试提供变通方案**：分析是否可以通过组合现有玩法或调整现有玩法的配置来近似实现用户需求，向用户说明变通方案的优缺点。
3. **请用户确认或指定**：让用户决定是使用变通方案、指定某个已有玩法来实现、还是确认需要新增玩法。
4. **如确认需要新增玩法**：在需求文档中新增一个专门的章节说明，包括：
   - 新增玩法的名称和建议的系统标识
   - 新增玩法的业务逻辑描述
   - 预期的参与流程和交互设计
   - 需要的配置项和数据结构
   - 与其他已有玩法的关联关系
   - 在文档的「4.2 玩法详细设计」中标注该玩法为**【新增玩法】**，提醒研发团队需要进行新功能开发

**玩法识别映射规则**（根据用户描述中的关键词/场景自动匹配）：

| 用户描述场景 | 对应系统玩法 | 系统标识 |
|------------|------------|---------|
| 点击预约、提前预约、预约新版本 | 预约 | preregister |
| 抽奖、转盘、随机奖励、消耗xx抽一次 | 抽奖 | lottery |
| 填问卷、调研、答题 | 问卷 | survey |
| 邀请好友、拉人、分享给好友注册/登录 | 邀请 | invite |
| 分享到朋友圈、转发、分享图片、每日分享、每周分享 | 分享 | share |
| 关注B站/抖音/微博/公众号/小红书 | 关注社媒 | follow |
| 订阅消息通知、微信提醒 | 微信订阅 | weixin_subscribe |
| 发弹幕、发评论、留言 | 评论/弹幕 | comment |
| 投票、选择支持、pick | 投票 | vote |
| 投稿、上传作品、提交内容 | 投稿 | submission |
| 为作品点赞、给投稿投票 | 投稿点赞 | submission_like |
| 组队、拉队伍、建小队 | 组队 | team |
| 完成任务、达成目标、累计登录X天、充值满X元、游戏内活跃值达到X | 任务 | quest |
| 开奖、兑奖、使用抽奖券 | 开奖 | lottery_draw |
| 签到、每日打卡 | 签到 | checkin |
| 兑换、用积分/代币换奖品 | 兑换 | exchange |
| 领礼包码、领CDK | 领取礼包码 | gift_code |
| 领激活码、获得游戏资格 | 领取激活码 | activation_code |

**玩法识别消歧规则（重要）**：

用户描述中可能同时匹配多个玩法关键词，必须按以下规则消歧：

1. **「每日/每周 + 分享」= share（分享），不是 quest（任务）**。「每日」「每周」描述的是参与周期（cycle），不是任务目标。只有描述中明确包含游戏内数据指标（如登录天数、充值金额、活跃值、对局次数、养猫数量等）才应识别为 quest。
2. **「每日签到」= checkin（签到），不是 quest（任务）**。签到是独立玩法，不要归类为任务。
3. **「邀请好友 + 分享链接」= invite（邀请），不是 share（分享）**。核心判断标准：如果目的是让好友完成登录/注册，则为 invite；如果目的仅是社交传播本身，则为 share。
4. **「投稿 + 投票」应拆分为两个独立玩法**：submission + vote，不要合并。
5. **quest（任务）的核心特征是：需要读取游戏内或系统内的数据指标来判定是否完成**。如「累计充值满100元」「登录游戏满7天」「活跃值达到200」等。单纯的用户操作行为（分享、签到、关注）不应识别为任务。

**在推荐玩法后，针对每个玩法逐一确认以下配置项**：

### 通用玩法配置项（每个玩法都需要确认）

- **玩法周期**（cycle）：无周期 / 每日 / 每周 / 每月
- **周期内参与次数上限**（cycle_limit）：0 为不限制
- **总参与次数上限**（limit）：0 为不限制
- **参与维度**（engage_account）：用户 ID / 角色 ID
- **玩法起止时间**：是否与活动整体时间一致？如不一致请说明

### 各玩法专属配置

根据玩法类型询问对应的关键配置。详细的玩法配置结构参见 `references/feature_configs.md`。

**关键联动关系**（自动检查并提醒用户）：
- 如果识别到**抽奖**，需确认消耗的道具是什么、奖池内容及权重
- 如果识别到**任务（quest）**，需确认任务目标类型（活跃值/登录天数/充值金额/社区互动/对局次数等）、完成值（completion_value）、是否关联组队。**quest 的奖励触发机制是通过 completion_value 判定任务完成，完成后产生参与记录并发放奖励，不要使用 REGULAR（按参与次数阶梯发放）来描述 quest 的奖励**。如果一个任务有多个阶梯目标，应拆分为多个独立的 quest 玩法，每个玩法设置各自的 completion_value。
- 如果识别到**组队**，需确认最大/最小队员数
- 如果识别到**开奖**，需确认关联的抽奖券道具和开奖时间窗口
- 如果识别到**投稿点赞**，需确认关联的投稿玩法
- 如果识别到**兑换**，但没有识别到产出该消耗道具的玩法，需主动提醒用户补充道具来源

## 阶段 4：道具与奖励设计

**首先梳理活动中涉及的所有道具**，道具类型包括：

| 道具类型 | 系统标识 | 说明 |
|---------|---------|------|
| 活动道具 | EVENT_ITEM | 仅在本活动中产出和消耗的代币型道具 |
| 游戏道具 | GAME_ITEM | 直接发放到游戏角色邮件的道具 |
| 实物奖品 | PHYSICAL_ITEM | 需用户填写收货地址 |
| 微信红包 | WEIXIN_HONGBAO | 需用户绑定微信账号 |
| 礼包码 | GIFT_CODE | 可在游戏内兑换的 CDK（唯一码/通兑码） |
| 激活码 | ACTIVATION_CODE | 激活游戏登录资格 |
| 抽奖券 | LOTTERY_TICKET | 用于开奖玩法的凭证 |
| 会员积分 | MEMBER_POINTS | 会员商城兑换积分 |
| 成长值 | GROWTH_VALUE | 会员成长值 |
| 空道具 | VOID_ITEM | "谢谢参与"场景 |

**需要明确的信息**：

1. **道具产出与消耗关系**：哪些玩法产出哪些道具？哪些玩法消耗哪些道具？请用户描述清楚活动的「经济循环」。
2. **奖励发放方式**：
   - **参与即发（WHENEVER）**：每次参与都发放
   - **达标发放（REGULAR）**：达到指定参与次数后发放
3. **特殊奖励说明**：
   - 涉及游戏道具时 → 需确认参与维度（用户 ID 还是角色 ID），并确认 GM 发放配置（邮件标题、内容、发件人、有效期）
   - 涉及实物奖品时 → 文档中需包含收货地址收集流程
   - 涉及微信红包时 → 文档中需包含微信账号绑定环节，且 EventConfig 中需配置 budget_id（**仅微信红包奖励需要 budget_id，其他奖励类型不需要**）
   - 涉及礼包码时 → 需确认是唯一码还是通兑码
4. **奖励限量配置**（如适用）：库存上限、每日限量、每人限量

## 阶段 5：风险、数据与补充

**风险点评估**（引导用户逐项确认）：

- **刷奖励/薅羊毛**：同一用户是否可能通过多设备、多账号重复领取？防护策略是什么？
- **黑产批量注册**：是否需要设置参与门槛（如账号注册天数、游戏等级等）？**注意：目前系统仅支持限制同一手机号的注册次数，不支持限制 IP 或设备的注册次数。如果用户提及 IP/设备限制，需主动告知这一限制。**
- **奖励超发**：涉及限量奖品时，库存控制策略是什么？
- **并发数据不一致**：高并发场景下的数据一致性保障（系统层面已通过分布式锁解决，但需求中仍需提及预期并发量级）

**数据统计需求**（引导用户勾选）：

- PV / UV（页面访问量和独立访客数）
- 活动参与人数 / 参与次数（按玩法维度）
- 奖励发放与消耗统计（各类道具产出/消耗流水）
- 分享 / 裂变数据（分享次数、通过分享链接带来的新增）
- 留存与回流数据（活动期间的登录留存、回流用户数）
- 其他自定义统计需求

**补充信息**（可选）：

- 是否需要配置短链接？**注意：仅游戏内嵌页（WebView）活动需要配置短链接，网页（H5）或微信小程序活动不需要配置短链接。** 如果活动平台不包含游戏内嵌页，则无需询问此项。
- 是否需要配置 GM 邮件（邮件标题、内容等）？

## 文档生成

所有阶段信息收集完毕后，生成 Markdown 格式的需求文档。

**简洁原则**：
- 避免在文档中重复信息。玩法总览表已包含框架配置（type、cycle、limit 等），各玩法详细设计中不再重复这些字段，只写专属配置和奖励。
- 玩法说明用一到两句话概括即可，不要过度展开。
- 前端交互流程用编号列表简述关键步骤，通常 3-5 步即可，不要拆得过细。
- 如果多个玩法共享相同的配置模式（如多个 follow 玩法仅平台不同），可以合并描述或用表格对比，不必各自重复完整小节。
- 道具经济循环部分用一张汇总表即可，不需要逐段文字描述每个产出来源。
- 风险点和数据统计按表格输出，不需要额外的段落展开。

## 流程图生成

在「用户参与流程」部分，使用 Figma 的 `generate_diagram` 工具生成流程图，并将流程图直接嵌入到文档对应位置。

**生成策略**：
1. 先生成一张**整体用户参与流程图**，嵌入到「3.1 整体流程概述」部分
2. 再为**每个玩法生成独立的交互流程图**，嵌入到「3.2 各玩法详细交互流程」中对应玩法的小节内

**整体流程图**应覆盖：
1. 用户访问活动页
2. 登录验证（根据平台不同分支）
3. 如涉及联运用户且需要游戏数据，包含 Combo ID 关联分支
4. 进入活动主页
5. 各玩法参与路径
6. 奖励领取
7. 活动结束

**各玩法流程图**应覆盖该玩法从用户点击到获得反馈的完整交互路径。

**嵌入方式**：每张流程图生成后，在文档中的对应位置使用 Figma generate_diagram 工具直接呈现给用户。

## 注意事项

- 文档语言为中文
- 玩法配置以表格形式呈现，不使用 JSON 格式
- 不包含接口设计内容（接口由研发单独出）
- 玩法间如有前置/解锁条件，需在文档中明确标注
- 涉及游戏道具发放时，必须说明参与维度和 GM 配置
- 涉及微信红包发放时，必须包含微信账号绑定环节，且 EventConfig 中需配置 budget_id
- 涉及非微信红包的游戏道具发放时，EventConfig 不需要配置 budget_id
- 涉及实物奖品时，必须包含收货地址收集流程
- 如参与用户包含联运用户或全渠道用户，且涉及游戏道具发放或游戏内数据读取，文档中必须包含 Combo ID 关联世游通行证的流程说明
- 防刷策略中，目前仅支持限制同一手机号的注册次数，不支持限制 IP 或设备。如用户提及 IP/设备限制需求，应主动告知此限制
- 仅游戏内嵌页（WebView）活动需要配置短链接，网页（H5）或微信小程序活动不需要
- 流程图使用 Figma generate_diagram 工具生成，整体流程图和各玩法流程图均需直接嵌入到文档对应位置
